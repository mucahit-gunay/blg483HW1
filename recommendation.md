# Production Deployment Recommendations

For deploying this crawler into a production environment, I would recommend the following next steps:

First, **replace SQLite with a dedicated database** such as PostgreSQL for the storage layer. While SQLite with WAL mode handles single-machine concurrency well, a production crawler needs horizontal read scaling, proper connection pooling, and concurrent write support that a client-server database provides. The search layer should also be upgraded to **Elasticsearch or OpenSearch** for full-text search with proper tokenization, stemming, and relevance tuning — the hand-rolled TF-IDF implementation works for demonstration but would not scale to millions of documents. Additionally, the in-memory crawl queue should be replaced with a **distributed message queue** like Redis or RabbitMQ to support multiple crawler workers across machines and provide durable, fault-tolerant job scheduling.

Second, the system should be **containerized with Docker** and deployed behind a reverse proxy (Nginx or Traefik) with proper health checks and auto-restart policies. For large-scale crawling, the architecture should shift to a **distributed worker model** where a coordinator service assigns URL batches to multiple crawler workers, each running as a separate process or container. This enables horizontal scaling and geographic distribution. Production concerns such as **structured logging** (ELK stack), **metrics collection** (Prometheus + Grafana), **rate limiting per target domain** (politeness policies beyond basic robots.txt), and **circuit breakers** for unreliable hosts should also be implemented. Finally, the crawler's HTTP layer should handle edge cases more robustly: proper redirect chain limits, content-length caps, timeout tuning, TLS certificate validation, and cookie/session management for sites that require them.
